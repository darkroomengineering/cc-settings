## 2. System map (per-area, from full reads)

### Installer & settings composition

The installer is a well-factored TS pipeline: setup.sh/setup.ps1 are thin bootstraps (ensure Bun, delegate to src/setup.ts) with genuine bash/PowerShell parity; composeSettings validates config/ fragments against the Settings zod schema and fails loudly on a bad fragment; all disk writes go through atomicWriteJson/atomicWriteString (tmp+rename); the settings.json merger (settings-merge.ts) is a clean per-key strategy table (permissions union, hooks union with deprecation-prune, env shallow-merge, statusLine reset-on-stale) with real regression-tested logic for the documented "Claude Code reorders hook JSON keys" problem via canonicalKey, and cmdRollback correctly guards against path-traversal/absolute-path tar entries. The light/full profile split is table-driven (PROFILE_MANIFEST) so dry-run/summary/install can't drift from each other. Where it falls short is at the seams between subsystems added in different eras: the code-intel-engine indirection (PR #71, meant to make `tldr`'s backing process swappable) was bolted onto an MCP-merge module whose core invariant — "existing entries in a shared key always win, that's the user's customization" — cannot distinguish cc-settings' own previous write from a genuine user edit, so the engine selection silently stops taking effect after the very first install. Similarly, the light-profile strip logic (which has to reverse-engineer "is this cc-settings-managed" from settings.json content) uses a shallower identity check than the sibling settings-merge.ts module, which already had to fix the same class of bug once. And the backup/cleanOldConfig pairing over-promises: the wipe is broad, the backup is narrow.

### Hot-path hooks & hook runtime

Area covers 10 hot-path hooks (src/hooks/*.ts) plus their shared runtime (hook-runtime.ts, hook-command.ts, hook-config.ts) and supporting libs (quota.ts, review-queue.ts, freeze.ts, status.ts/-types.ts, colors.ts), wired via config/40-hooks.json. Overall the fail-open discipline is genuinely sound and consistently applied: every hook wraps its body in runHook()/try-catch, blockDecision() is the single shared block-grammar, readState/writeState use atomic tmp+rename, and quota.ts validates its cache with zod (RateLimitsCacheSchema/QuotaSteerStateSchema) before trusting it. Quota-steer thresholds (60/85 five-hour, 65/85 seven-day, 10-min stale cutoff, 30-min critical re-remind) match docs/codex-bridge.md exactly. The freeze feature's documented Bash-bypass limitation (skills/freeze/SKILL.md explicitly discloses "shell writes (Bash) are not gated") is transparent, not a hidden gap, so I did not file it. Similarly the "Plan"/"Explore" capitalized entries in review-queue.ts's READ_ONLY_AGENTS turned out to be Claude Code's own built-in SubagentStart matcher names (confirmed via docs/hooks-reference.md's SubagentStart/SubagentStop matcher table) coexisting with this repo's lowercase custom agents — not a bug, and I discarded my initial suspicion after checking. Config/40-hooks.json wiring is largely coherent with what each hook's code expects (matchers align with the fields each hook reads), with one exception noted below. The most consequential surviving finding is in the statusline hot path, where two of four parallel git spawns bypass the codebase's own git-timeout convention (lib/git.ts's runGit/runProcessFull always pass `timeout` + SIGKILL) and could hang the per-turn statusline refresh indefinitely, directly undercutting the file's own "never a blank line" hardening claim.

### One-shot scripts

Read all 34 files in src/scripts/ plus the 8 assigned src/lib/ modules. Overall the one-shot-script layer is competently built: process spawns are timeout-bounded (git.ts, tsc.ts), JSON state is schema-validated on read (checkpoint.ts, knowledge-index.ts), atomic-rename is used correctly for "latest" symlinks (artifact-store.ts), and several scripts show real care about a specific footgun — explicit `Number.isNaN(...)` guards against `parseInt(x) || default` silently reviving a default when the user explicitly passed "0" (checkpoint.ts cmdClean, handoff.ts cmdClean, prune-mcp-auth-cache.ts). That same care is inconsistently applied, though, and the inconsistency is where most defects live: checkpoint.ts's artifact store is scoped per-project and collision-avoids same-second saves; handoff.ts (structurally the same "timestamped artifact + latest symlink" pattern, built on the same artifact-store.ts helpers) is NOT project-scoped at all, contradicting its own skill doc's example filenames. Several scripts are hooks with fail-open contracts that are honored correctly (check-docs-before-install.ts, notify.ts, post-failure.ts, stop-summary.ts all swallow errors as documented). The main gaps found are: cross-project state bleed in the handoff store, the automatic (hook-driven) handoff path never actually populating the content it promises, a dead-on-arrival regex in check-docs-before-install.ts, proof.ts discarding the one thing you need on failure (gate output), and one repeat of the falsy-zero parsing bug in log-bash.ts that the codebase explicitly guards against everywhere else. Exit-code behavior is also inconsistent between the two sibling CLIs (checkpoint.ts vs handoff.ts) for the same "unknown subcommand" case, which is exactly the kind of thing an agent parsing exit codes would misread as success.

### Supply-chain defense & command-execution surfaces

Read all 11 assigned files in full plus config/40-hooks.json, hook-runtime.ts, and grepped for DEPRECATED_COMMAND_PATTERNS/TOOL_INPUT_command to verify cross-file claims. The three-layer supply-chain defense (hooks fingerprint, src content manifest, engine pin) is internally coherent and its SessionStart hook is genuinely fail-open (each check independently try/caught). The hook auditor's classification logic (trusted/unknown/stale/suspicious) is sound: it correctly requires content-hash verification (not path-shape) for "trusted", checks malware signatures unconditionally first, and the compound-command classifier correctly uses `.every()` (not `.some()`) so a `managed || evil` chain can't launder to trusted. safety-net.ts's rm/git/checkout parsers are well-reasoned with documented residual gaps (quoted executables) that SECURITY.md explicitly scopes as advisory-only. Codex bridge (codex.ts, codex-run.ts) uses array-form Bun.spawn with `--` to end option parsing, avoiding shell-injection into the prompt — solid. However, three real gaps surfaced: (1) the fingerprint's hash scope is `settings.json`'s `hooks` key only — sibling keys like `mcpServers` (arbitrary command execution on every session, a stronger persistence primitive than a hook) or `permissions` are entirely outside all three integrity checks, and SECURITY.md's "what we don't cover" list never names this; (2) config/30-permissions.json wholesale-allows several git subcommands whose dangerous sub-forms have no deny-list counterpart, while safety-net.ts (documented as advisory/fail-open, NOT the enforcement boundary per SECURITY.md itself) is the only thing standing between the user and running them; (3) log-bash.ts logs every Bash command (not just blocked ones) through a weaker redaction function than safety-net's, missing GitHub/AWS token and password= patterns entirely.

### Schemas, linters & upstream scanner

Scope: src/schemas/* (10 zod files), schemas/*.schema.json (generated), the 6 lint-*.ts linters, frontmatter.ts/frontmatter-validate.ts, and src/upstream/scan.ts + version-delta.ts/engine-pin.ts/code-intel-engine.ts. Verified `bun run schemas:emit && git diff schemas/` produces zero diff — the generated JSON Schemas are currently fresh against the zod sources, and `bun run src/upstream/scan.ts` executes cleanly end to end (all `.value`/`.unwrap().value` ZodLiteral accessors work as documented). The loose-vs-strict policy is applied consistently and is well-reasoned: files Claude Code itself writes (ClaudeJson, Settings) are `z.looseObject` for forward-compat, with a dedicated top-level-key-only typo guard in tests/schemas.test.ts; agent/skill/profile frontmatter are also loose by design for ecosystem velocity, with install-time (non-fatal) and CI (fatal) validation layered via frontmatter-validate.ts and the individual lint-*.ts scripts respectively. The MCP and Skill "shape guard" pattern (reject cross-shape key mixing before the union strips anything) is a genuinely sound design that correctly catches the stdio/http mixing and command/mcp mixing cases I traced by hand. Running the upstream scanner live surfaced real, current drift it is designed to catch (schema has `advisorModel`, manifest doesn't) — proving the tool works for its stated purpose, but also proving that purpose is narrower than the name suggests: it only ever diffs two hand-maintained local artifacts against each other (manifest.json vs zod schemas) plus an npm version number: it has no path to detect a genuinely new upstream Claude Code feature that a human hasn't yet transcribed into either file, and the CI job for it is explicitly dry-run/non-blocking so this class of drift can persist indefinitely. The main defect cluster is at the loose-object boundary and doc/schema/linter parity: AgentFrontmatter is missing three fields (`skills`, `background`, `allowedTools`) that are actively documented as agent capabilities elsewhere in the repo, so typos in those fields get zero protection despite frontmatter-validate.ts's docstring explicitly promising typo protection; the security-motivated angle-bracket scan exists only for skills, not agents/profiles, despite the same "injected into system prompt" rationale; and kebab-case name validation is implemented with three subtly different regexes across schemas/linters with no shared constant.

### Skill library

Read all 37 skill directories in full (skills/README.md is documentation, not a skill; skills/nuclear-review/references/ is an opt-in example script). The library is disciplined overall: `bun run lint:skills` passes clean (0 errors/warnings); `ACTIVE_SKILLS` in src/lib/managed-skills.ts matches skills/ on disk exactly (37=37); MANUAL.md's "All Skills" table also matches disk exactly (37=37, including the newly added `harvest`); every `bun run X` command a skill tells the model to run exists in package.json; every `~/.claude/src/scripts/*.ts` path referenced (checkpoint.ts, freeze.ts, handoff.ts, codex-run.ts, new-skill.ts, review-batch.ts, proof.ts) exists with matching subcommands; every `agent:` frontmatter reference (explore, planner, reviewer, tester, maestro) points to a real agents/*.md file; cross-references between skills (context-doc↔explore↔test via DOMAIN-AWARENESS.md, fix↔ship, refactor↔implementer, qa↔verify, nuclear-review↔adversarial-audit↔zero-tech-debt) are current and explicit, and I found no live skill referencing a retired/merged skill by its old slash-command name — every hit on a tombstoned name (`/context`, `/init`, `/lenis`, `/prd`, `/docs`) is a deliberate "formerly X" or "NOT the same as native X" disambiguation, not stale routing. Trigger-phrase space is also well-managed: near-neighbor pairs (fix/verify, review/nuclear-review/adversarial-audit, checkpoint/handoff, refactor/zero-tech-debt) each carry an explicit "use X instead when Y" line. The true current skill count is 37, confirmed three independent ways (disk, ACTIVE_SKILLS, MANUAL.md table). The defects that do survive are concentrated in two spots: agent/tool-capability mismatches where a skill's documented workflow needs a tool its bound agent or its own `allowed-tools` frontmatter doesn't grant, and doc-count drift around the skill total itself (ironic given the library's own doctrine about drift).

### Agent, profile & rule definitions

Area covers agents/ (10 role agents), profiles/ (6 stack profiles), rules/ (10 path-conditioned rule files + README). Most frontmatter is internally sound: tool lists pass the schema's loose-string validation (src/schemas/agent.ts accepts any string for `tools`), effort/permissionMode values are all legal enum members, and CLAUDE-FULL.md's delegation prose correctly names real agents (explore, planner, implementer, reviewer, tester, scaffolder, deslopper, security-reviewer, maestro, codex-verifier). Profile skills/agent references (build, component, hook, lighthouse, qa, orchestrate) all resolve to real skills. Where the area breaks down is at the seams between docs and the frontmatter they describe: docs/frontmatter-reference.md's own "All Agents in cc-settings" table hasn't been updated for the Sonnet-5 downshift that agents/implementer.md, agents/reviewer.md, and docs/agent-models.md already reflect; docs/agent-models.md documents a model pin for an `oracle` agent that has no agents/oracle.md file (oracle is actually a skill routed to `explore`/sonnet); and the isolation:worktree feature has a documented gotcha (fresh worktrees branch off origin/<default-branch>, losing uncommitted/unpushed work) that reviewer.md and security-reviewer.md walk straight into via their own initialPrompt without applying the documented mitigation. A schema/docs split also exists on `isolation`: the schema enum accepts only `worktree`/`remote`, but the mitigation docs tell users to set `isolation: "head"`, which fails frontmatter validation. Team-mode workflows in deslopper.md and maestro.md also reference tool calls (TeamDelete; the whole Team/Task toolset) absent from those agents' own `tools:` arrays.

## 4. Design tensions

The five deepest structural issues. These are about the approach, not a line; the line findings above are mostly symptoms of these.

### T1 — Block-level "user wins" merging cannot tell a user's edit from cc-settings' own stale output

The single largest cluster of findings (H8, H9, M — array defaults, M — light-profile mixed groups, L — stripManagedSettings identity check) plus two previously-known warts (the hook-rename stale-audit flag, the nested-config-defaults gap) all reduce to one missing concept: **provenance**. The merger preserves whatever exists on disk, so anything cc-settings wrote yesterday becomes immovable "user customization" today. Every new managed surface (engine pins, nested defaults, renamed hooks) re-collides with this.
**Alternative to weigh:** a per-key provenance manifest — at install time, record a hash of every value cc-settings writes (the hooks fingerprint already does exactly this for one block). On the next install: value matches our last write → ours, safe to update; value differs → genuinely user-edited, preserve. One mechanism retires the whole bug class and the hook-rename tombstone logic.

### T2 — The supply-chain defense narrates a broader perimeter than it patrols

SECURITY.md's threat model is settings.json tampering, but the fingerprint, the content-manifest audit, and iterCommandHooks all read only `settings.hooks` (H12). Meanwhile `env` (e.g. `ANTHROPIC_BASE_URL`, `apiKeyHelper`-adjacent vars), `permissions.allow`, and `mcpServers` are equally executable-by-consequence and unwatched — and the permission fragment itself allows wholesale `git push`/`checkout` with no deny counterparts (M) and leaves Edit/MultiEdit as a documented bypass of the Write deny on settings.json (M).
**Alternative to weigh:** either extend the fingerprint to the full managed surface of settings.json (cheap: it is one more hash input), or shrink the claim — a scoped SECURITY.md that says exactly "hooks block only" is honest and ships today.

### T3 — Capability contracts between skills/agents and their tools are enforced nowhere

H2 (maestro's Team-less "Agent Teams Mode"), H13 (oracle compare needs AskUserQuestion its agent lacks), H14 (qa's Fresh-Eyes Gate needs a tool its own allowed-tools excludes), and the deslopper teardown step (M) are all the same defect: prose in a definition file promises a workflow; frontmatter denies the tools it needs; no linter compares the two. These fail only at runtime, in someone else's session, in a way the user reads as "the agent is being lazy".
**Alternative to weigh:** a lint pass that extracts tool names referenced in the body of every skill/agent/profile and diffs them against the frontmatter grant. The repo already has six bespoke linters; this one would have caught 4 findings.

### T4 — Hand-maintained doc tables are a second source of truth the repo already knows how to generate

The docs area is the largest finding bucket (hooks-reference missing 4 live hooks, model table naming a nonexistent agent, three different skill counts, frontmatter-reference stale on models/effort/context-modes, MANUAL's phantom Setup event and phantom Sanity core server). The repo already solves this pattern for JSON Schema (`schemas:emit` from zod).
**Alternative to weigh:** extend the emit step — hooks-reference's "Configured Hooks" table from `config/40-hooks.json`, the model-routing table from `agents/*.md` frontmatter, skill counts from `ls skills/`. Drift then becomes a CI failure, not an audit finding.

### T5 — Sibling modules re-solve solved problems divergently

Three secret-redaction functions with three different pattern sets (the weakest one guarding the most sensitive log), three kebab-case regexes, two "is this managed" checks, two exit-code conventions for the same CLI shape (checkpoint vs handoff), an inline re-implementation of the frontmatter regex next to the module that exports it. None is individually serious; together they mean every fix must be discovered N times.
**Alternative to weigh:** a `src/lib/primitives/` layer (redaction, slug regex, managed-identity, CLI exit contract) with a lint rule against local re-implementations — the same consolidation move the skill library already went through in May.

## 5. Open questions (maintainer input required)

1. **Is `settings.json`'s `mcpServers` block a real surface or a vestige?** H9's fix direction depends on it: either the engine-mutated map must flow into both files, or `~/.claude.json` should be declared the single MCP source of truth and the settings.json block dropped from the merge. `docs/settings-reference.md` currently documents it as first-class.

If that is an accepted risk, SECURITY.md should say so; if not, it is an M-severity gap to close.
3. **What is the light tier's actual hook promise?** "No hooks except statusLine" (then the mixed-group strip must be surgical, per M finding) or "no *cc-settings* hooks" (then current behavior is close and the docs overstate)?

5. **Does anything still need the installer to provide `jq`?** No runtime hook shells out to it anymore, but team runbooks (including two team-knowledge notes) give `jq` one-liners as remediation steps. Keep-and-document, or drop the forced install?
