# write-a-skill — scaffold a new cc-settings skill

cc-settings has 38+ skills. The native `Skill` tool auto-invokes them based on the `description` frontmatter — so the description is **the only thing the agent sees** when deciding whether to load yours. That's where most of the work goes.

## Process

1. **Gather requirements.** Ask the user:
   - What task does the skill cover?
   - What conversation phrases should auto-invoke it?
   - Which existing skills does it complement or replace? (Avoid overlap with `/fix`, `/build`, `/explore`, `/review`, etc.)
   - Does it need supporting reference files, or is one `SKILL.md` enough?
   - Any executable scripts? Skills can shell out via `bun "$HOME/.claude/src/scripts/<name>.ts"`.

2. **Draft the skill.** Create `skills/<name>/SKILL.md`. Match the cc-settings convention:

   ```md
   ---
   name: skill-name
   description: |
     Use when:
     - User says "<phrase 1>", "<phrase 2>"
     - <situational trigger>
     - <another trigger>

     <One-paragraph summary of what the skill does, what it produces,
     and how it relates to other cc-settings skills.>
   ---

   # skill-name

   <Body — see structure below>
   ```

3. **Register it.** Add `"<name>"` to `MANAGED_SKILLS` in `src/setup.ts` (alphabetical order).

4. **Document it.** Add a row to `MANUAL.md` in the appropriate section (Daily Workflows / Quality / Planning / Maintenance / etc.) and the "All Skills" table at the bottom.

5. **Validate.** `bun test && bun run typecheck`. Tests should still pass — the schemas in `src/schemas/skill.ts` validate frontmatter shape.

## Frontmatter rules

- `name` — kebab-case. Must match the directory name.
- `description` — multi-line `description: |` block. The cc-settings convention starts with bullet `Use when:` triggers and ends with a one-paragraph summary.
- Max ~1024 chars total. The description sits in every agent's system prompt; verbose descriptions burn context across the fleet.
- Write triggers in the user's voice ("user says", "user wants"), not the agent's.
- Be specific. "Helps with code" lets the agent confuse this with every other code skill. "Triages GitHub issues through a label-based state machine" doesn't.

## Body structure

Match the existing skills (`skills/fix/SKILL.md`, `skills/cc-sync/SKILL.md`, `skills/consolidate/SKILL.md` are good models):

- Open with a one-sentence statement of what the skill produces or enforces.
- Phase headings (`## Phase 1 — …`) for multi-step workflows.
- Concrete commands and bash snippets where applicable.
- Approval gates for destructive or judgment-call work — mark them clearly with **Stop. Present the plan.** style headings.
- Close with "What this skill does NOT do" if there's a real boundary worth flagging.

## When to add reference files

Split content across additional files in the skill's directory only when:

- `SKILL.md` exceeds ~150 lines and has distinct sub-domains
- A format spec or template is reused (e.g., `CONTEXT-FORMAT.md`, `ADR-FORMAT.md`)
- A consumer-rules doc is shared with sibling skills (e.g., `DOMAIN-AWARENESS.md`)

Reference from `SKILL.md` with a relative link: `[CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md)`.

## When to add scripts

Add a TypeScript helper in `src/scripts/` when:

- The operation is deterministic (parse a file, format output, validate input)
- The same logic would otherwise be regenerated by the agent every time
- Errors need explicit handling that the prose can't enforce

Reference from the skill: `bun "$HOME/.claude/src/scripts/<name>.ts"`.

## Review checklist

Before committing:

- [ ] `description` includes specific `Use when:` triggers
- [ ] No overlap with existing skills (or overlap is intentional and noted)
- [ ] Frontmatter passes `bun test`
- [ ] Skill registered in `MANAGED_SKILLS` (alphabetical)
- [ ] Added to MANUAL.md
- [ ] Concrete examples or commands in the body
- [ ] Approval gates marked for destructive operations
- [ ] No time-sensitive info (versions, dates) in the body
